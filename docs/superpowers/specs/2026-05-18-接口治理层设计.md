# 接口治理层设计

## 目标

在现有 `Koa + Sequelize + MySQL` 后端基础上，补齐一层更稳定的接口治理能力，让 `users`、`malls`、`accounts` 这三个核心 CRUD 模块在分页、筛选、异常语义、审计日志和接口说明上保持一致，方便前端长期联调和后续部署。

## 当前问题

1. 列表接口的筛选字段不统一，`users` 和 `malls` 主要用 `name`，`accounts` 用 `username`，前端接入时需要分别记忆。
2. 分页返回结构虽然都有 `list` 和 `count`，但没有把当前页和页大小一起返回，前端状态同步要自行拼装。
3. 登录登出已经有日志，但用户、商品、账号的增删改没有统一审计记录，排查问题时链路不完整。
4. 错误码大体可用，但缺少统一语义约束，业务层和控制层后续容易各写各的。
5. 目前缺少一份中文接口文档，后续联调、复盘、部署时沟通成本高。

## 设计原则

1. 尽量不改现有前端已依赖的基础响应壳结构，继续保留 `code`、`message`、`data`、`requestId`。
2. 在兼容旧参数的前提下引入统一查询参数，避免一次性破坏前端当前调用。
3. 审计日志先做文件日志，不额外引入数据库审计表，控制复杂度。
4. 优先复用现有 `controller -> service -> model` 结构，不做大规模架构调整。
5. 文档全部使用中文，便于你后续自己维护。

## 统一接口约定

### 列表查询参数

三个列表接口统一支持以下参数：

- `page`：页码，默认 `1`
- `pageSize`：每页条数，默认 `10`
- `keyword`：统一搜索关键字，默认空字符串

兼容策略：

- `users`：继续兼容 `name`
- `malls`：继续兼容 `name`
- `accounts`：继续兼容 `username`

解析规则：

- 优先读取 `keyword`
- 如果 `keyword` 为空，再回退到旧字段
- `page` 和 `pageSize` 做数字归一化，非法值回退默认值

### 列表响应结构

列表接口统一返回：

```json
{
  "code": 200,
  "message": "success",
  "data": {
    "list": [],
    "count": 0,
    "page": 1,
    "pageSize": 10
  },
  "requestId": "..."
}
```

这样前端分页组件可以直接使用后端返回值，不再自己维护页码回显。

## 异常语义规范

统一语义如下：

- `400`：参数错误、格式错误、业务前置条件不满足
- `401`：未登录、令牌无效、令牌缺失
- `403`：已登录但无权限，或账号被禁用
- `404`：目标资源不存在
- `409`：资源冲突，例如账号已存在
- `500`：未捕获服务器异常

消息策略：

1. 新增和重构到的校验消息统一改为清晰中文。
2. 不强行改动所有旧模块的所有文案，但本次涉及到的 `users`、`malls`、`accounts` 统一到中文语义。
3. 错误返回仍通过 `ctx.error(message, code)` 输出，避免打破现有中间件。

## 审计日志设计

新增统一审计日志工具，记录这些信息：

- `requestId`
- `operator`
- `role`
- `module`
- `action`
- `targetId`
- `result`
- `detail`

记录范围：

- `accounts`：创建、修改状态/角色、重置密码、删除
- `users`：创建、修改、删除
- `malls`：创建、修改、删除
- `permission`：保留现有登录登出日志

输出策略：

1. 继续基于现有 `log4js`
2. 增加 `audit` 分类，单独落日志文件
3. 成功和失败都写入，失败至少记录模块、动作、操作者和原因

## 代码结构调整

计划新增两个通用工具：

1. `backend/utils/pagination.js`
   - 负责归一化 `page`、`pageSize`、`keyword`
   - 负责构造统一列表响应数据

2. `backend/utils/auditLog.js`
   - 负责统一输出审计日志
   - 控制器只需要传模块、动作、目标和结果

本次不引入新的 service 抽象层，不做泛 CRUD 基类。

## 测试策略

新增接口测试覆盖：

1. `users` 列表支持 `keyword`，并返回 `page`、`pageSize`
2. `malls` 列表支持 `keyword`，并返回 `page`、`pageSize`
3. `accounts` 列表支持 `keyword`
4. 参数校验失败时返回中文错误消息与正确状态码
5. 权限错误仍保持 `401/403`

审计日志本次不做文件内容断言测试，避免测试和日志文件系统强耦合；只验证主流程不被日志写入影响。

## 文档范围

新增一份中文接口文档，至少覆盖：

- 鉴权方式
- 通用响应结构
- `users`、`malls`、`accounts` 的列表/新增/修改/删除接口
- 统一分页筛选参数说明
- 常见错误码说明

## 成功标准

满足以下条件即可进入下一提交点：

1. 三个列表接口都支持统一的 `keyword/page/pageSize`
2. 三个列表接口都返回 `list/count/page/pageSize`
3. `users`、`malls`、`accounts` 的增删改具备审计日志
4. 本次涉及的校验与资源不存在消息改为清晰中文
5. 新增测试通过
6. 中文接口文档可直接用于后续联调
